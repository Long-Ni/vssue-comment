# 1. 各分支命名及其功能

# 1.1. 永久分支

# 1.1.1. master 分支

# 1. 命名: master
# 2. 规则:
  • master 分支代码与线上保持一致。
  • master 分支每个 commit 都有对应的 tag ，用于管理版本。
  • master 分支接受的每一个合并，都对应一个经过严格测试的主管稳定的线上版本。

# 1.1.2. develop 分支

# 1. 命名: develop
# 2. 必要规则:

  • develop 分支基于 master 分支创建。

  • develop 分支是自动构建分支，它总是保有当前即将发布或是已经发布的代码，是确定下一次将要通过 release 分支合并到 master 上的代码。

  • develop 分支不直接合并到 master 分支， 而是通过 release 分支合并到 master 分支。

  • develop 分支在 git flow 中承担了最为复杂与重要的任务。

  • develop 分支是代码的同步中心，其他任何分支，都是经过 develop 分支完成代码同步的。

# 1.2. 临时分支

# 1.2.1. feature 分支

# 1. 命名: 以 “feature/” 开头
# 2. 必要规则:

  • feature 分支基于 develop 分支创建。

  • feature 分支是进行新特性开发的分支。

  • feature 分支做完后，确定为当前上线（下个版本）的特性后，便可合并回 develop 分支。

  • feature 分支做完后，确定不是当前上线（下个版本）的特性，不要合并到 develop 分支。

# 3. 可选规则:

  • feature 分支合并回 develop 分支后，建议删除其分支。

# 1.2.2. release 分支

# 1. 命名: 以 "release/" 开头
# 2. 必要规则:
  • release 分支基于 develop 分支创建。
  • release 分支是用于特性发布，工程配置，修改次要 bug，stage 环境测试等。其他开发人员可以基于 develop 分支开发新的特性。
  • release 分支一旦打开之后，不要从 develop 分支上合并新的改动到 release 分支。
  • release 分支发布前，该特性引入的常规 bug 在本分支上修改。
  • release 分支完成后（发布时），合并 release 分支到 master 和 develop 分支。
  • release 分支合并回 mastere 分支时，在master 分支打上 tag （大版本 ）记住 Release 的版本号。
# 3. 可选规则:

  • release 分支合并回 master 分支和 develop 分支后，建议删除其分支。

# 1.2.3. hotfix 分支

# 1. 命名: 以 "hotfix/" 开头
# 2. 必要规则:
  • hotfix 分支基于 master 分支并选取特定的 tag 创建。
  • hotfix 分支是为解决线上严重问题而生，用于生产环境程序修复。
  • hotfix 分支在完成测试具备上线条件后，在 hotfix 分支上完成版本的发布。
  • hotfix 分支完成后需要合并回 master 和 develop 分支。
  • hotfix 分支完成后，先合并到 master 分支，再合并到 develop 分支。这两个合并步骤是不可分的，是一个十五。也可称为同时合并到 master 分支和 develop 分支。
  • hotfix 分支合并回 master 分支时，在 master 分支上打上 tag（小版本）。
# 3. 可选规则:

  • hotfix 分支合并回 master 分支和 develop 分支后，建议删除其分支。

# 1.2.4. bugfix 分支

# 1. 命名: 以 "bugfix/" 开头
# 2. 必要规则:
  • bugfix 分支基于 develop 分支创建。
  • hotfix 分支用于版本发布后发现的 bug 修复，或者因其他特性引入的 bug 修复。
# 3. 可选规则:
  • hotfix 分支合并回 develop 分支后，建议删除其分支。

# 1.3. 分支操作注意事项

  • 只能对自己的私有分支做 rebase 操作。
  • 凡是涉及到几大共有分支上的合并，必须使用 merge，禁止使用 rebase。严格禁止共有分支上的提交 id 变更。
